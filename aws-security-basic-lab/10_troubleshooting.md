## 概要

このドキュメントでは、AWS Security Basic Lab の構築中に発生した問題と、その原因・対応方法をまとめる。

対象範囲は、AWS Config、S3、CloudWatch Alarm などである。

---

## 1. AWS Config作成時に既存のS3バケットを選択したが権限がなかった

### 発生した問題

AWS Configを有効化する際、設定履歴の配信先として既存のS3バケットを選択したが、権限不足により設定を完了できなかった。

### 原因

AWS ConfigがS3バケットへ設定履歴やスナップショットを書き込むには、対象S3バケットにAWS Configからの書き込みを許可するバケットポリシーが必要である。

既存のS3バケットでは、AWS Config用の権限設定が不足していたため、配信先として利用できなかった。

### 対応方法

既存バケットの権限を修正するのではなく、AWS Config用のS3バケットを新規作成した。

```text
対応内容:
- AWS Config用のS3バケットを新規作成
- Block Public Accessを有効化
- AWS Configの配信先として新規バケットを指定
```

## 2. CloudWatch Alarm作成時に名前を間違えた

### 発生した問題

CloudWatch Alarm作成時に、アラーム名を間違えて作成してしまった。

作成後に名前を変更しようとしたが、CloudWatch Alarmの名前は後から変更できなかった。

### 原因

CloudWatch Alarmでは、作成済みアラームの名前を直接変更できない。

アラーム名を変更したい場合は、新しい名前でアラームを作り直す必要がある。

### 対応方法

誤った名前のCloudWatch Alarmを削除し、正しい名前で新しく作成し直した。

```text
対応内容:
- 誤った名前のCloudWatch Alarmを削除
- 正しい命名ルールでCloudWatch Alarmを再作成
- 対象メトリクス、しきい値、評価期間を再設定
```
