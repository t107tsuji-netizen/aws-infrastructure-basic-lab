
# 1. VPCを作成する

VPCは、AWS上に作成する専用の仮想ネットワークである。

AWSマネジメントコンソールで、以下の順番に移動する。

```text
AWSコンソール
↓
VPC
↓
お使いのVPC
↓
VPCを作成
```

以下の内容を設定する。

```text
作成するリソース: VPCのみ
名前タグ: my-vpc-01
IPv4 CIDR: 10.0.0.0/16
IPv6 CIDRブロック: IPv6 CIDRブロックなし
テナンシー: デフォルト
```

設定後、`VPCを作成` を選択する。

---

# 2. パブリックサブネットを作成する

踏み台サーバーを配置するため、パブリックサブネットを作成する。

AWSマネジメントコンソールで、以下の順番に移動する。

```text
VPC
↓
サブネット
↓
サブネットを作成
```

以下の内容を設定する。

```text
VPC ID: my-vpc-01
サブネット名: public-subnet-1a
アベイラビリティーゾーン: ap-southeast-1a
IPv4サブネットCIDRブロック: 10.0.1.0/24
```

設定後、`サブネットを作成` を選択する。

```text
VPC: 10.0.0.0/16
└─ public-subnet-1a: 10.0.1.0/24
```

# 4. インターネットゲートウェイを作成する

踏み台サーバーをインターネットへ接続するため、インターネットゲートウェイを作成する。

AWSマネジメントコンソールで、以下の順番に移動する。

```text
AWSコンソール
↓
VPC
↓
インターネットゲートウェイ
↓
インターネットゲートウェイを作成
```

以下の内容を設定する。

```text
名前タグ: my-igw
```

設定後、`インターネットゲートウェイを作成` を選択する。

---

## 4.1 インターネットゲートウェイをVPCへアタッチする

```text

↓

↓
my-vpc-01を選択
↓
インターネットゲートウェイのアタッチ
```

以下の状態になっていることを確認する。

```text

```

# 5. パブリックサブネット用ルートテーブルを作成する

踏み台サーバーを配置するパブリックサブネット用のルートテーブルを作成する。

AWSマネジメントコンソールで、以下の順番に移動する。

```text
VPC
↓
ルートテーブル
↓
ルートテーブルを作成
```

以下の内容を設定する。

```text
名前: public-rt01
VPC: my-vpc-01
```

設定後、`ルートテーブルを作成` を選択する。

作成直後は、以下のルートが自動的に登録されている。

```text
送信先: 10.0.0.0/16
ターゲット: local
```

---

## 5.1 インターネットゲートウェイへのルートを追加する

作成した `public-rt01` に、インターネットゲートウェイへのルートを追加する。

```text
public-rt01を選択
↓
ルート
↓
ルートを編集
↓
ルートを追加
```

以下の内容を設定する。

```text
送信先: 0.0.0.0/0
ターゲット: Internet Gateway
対象: my-igw
```

設定後、`変更を保存` を選択する。

最終的なルートは以下となる。

```text
10.0.0.0/16 → local
0.0.0.0/0 → my-igw
```

---

## 5.2 パブリックサブネットを関連付ける

作成したルートテーブルを、パブリックサブネットへ関連付ける。

```text
public-rt01を選択
↓
サブネットの関連付け
↓
サブネットの関連付けを編集
↓
public-subnet-1aにチェック
↓
関連付けを保存
```

# 7. 踏み台サーバー用セキュリティグループを作成する

自宅PCから踏み台サーバーへSSH接続するためのセキュリティグループを作成する。

AWSマネジメントコンソールで、以下の順番に移動する。

```text
AWSコンソール
↓
EC2
↓
ネットワーク＆セキュリティ
↓
セキュリティグループ
↓
セキュリティグループを作成
```

以下の内容を設定する。

```text
セキュリティグループ名: bastion-sg
説明: Allow SSH from my IP
VPC: my-vpc-01
```

インバウンドルールに以下を追加する。

```text
タイプ: SSH
プロトコル: TCP
ポート範囲: 22
ソース: マイIP
```

設定例は以下である。

| タイプ | プロトコル | ポート | ソース |
|---|---|---:|---|
| SSH | TCP | 22 | 自宅PCのグローバルIPアドレス `/32` |

アウトバウンドルールは初期設定のままとする。

# 8. 内部Linux用セキュリティグループを作成する

踏み台サーバーから内部LinuxへSSH接続するためのセキュリティグループを作成する。

AWSマネジメントコンソールで、以下の順番に移動する。

```text
AWSコンソール
↓
EC2
↓
ネットワーク＆セキュリティ
↓
セキュリティグループ
↓
セキュリティグループを作成
```

以下の内容を設定する。

```text
セキュリティグループ名: private-linux-sg
説明: Allow SSH from bastion server
VPC: my-vpc-01
```

インバウンドルールに以下を追加する。

```text
タイプ: SSH
プロトコル: TCP
ポート範囲: 22

```

設定例は以下である。

| タイプ | プロトコル | ポート | ソース |
|---|---|---:|---|
| SSH | TCP | 22 | `bastion-sg` |

ソースには、自宅PCのIPアドレスではなく、踏み台サーバーに設定する `bastion-sg` を指定する。

アウトバウンドルールは初期設定のままとする。

# 9. キーペアを作成する

踏み台サーバーと内部LinuxへSSH接続するためのキーペアを作成する。

AWSマネジメントコンソールで、以下の順番に移動する。

```text
EC2
↓
ネットワーク＆セキュリティ
↓
キーペア
↓
キーペアを作成
```

以下の内容を設定する。

```text
キーペア名: my-ec2-key
キーペアのタイプ: RSA
プライベートキーファイル形式: .pem
```

キーペアを作成すると、以下のファイルがPCへダウンロードされる。

```text
my-ec2-key.pem
```

秘密鍵はSSH接続で使用するため、安全な場所へ保存する。

# 10. 踏み台サーバーを作成する

パブリックサブネットへ踏み台サーバーを作成する。

AWSマネジメントコンソールで、以下の順番に移動する。

```text
AWSコンソール
↓
EC2
↓
インスタンス
↓
インスタンスを起動
```

---

## 10.1 名前とAMIを設定する

以下の内容を設定する。

```text
名前: test-amazonlinux2023-bastion
AMI: Amazon Linux 2023
```

Red Hat Enterprise Linuxを使用する場合は、RHELのAMIを選択する。

---

## 10.2 インスタンスタイプを選択する

以下の内容を設定する。

```text
インスタンスタイプ: t3.micro
```

起動画面で、利用するAWSアカウントの無料利用枠対象であるかを確認する。

---

## 10.3 キーペアを選択する

作成したキーペアを選択する。

```text
キーペア名: my-ec2-key
```

---

## 10.4 ネットワークを設定する

以下の内容を設定する。

```text
VPC: my-vpc-01
サブネット: public-subnet-1a
パブリックIPの自動割り当て: 有効
ファイアウォール: 既存のセキュリティグループを選択
セキュリティグループ: bastion-sg
```

---

## 10.5 ストレージを設定する

以下の内容を設定する。

```text
容量: 8 GiB
ボリュームタイプ: gp3
```

設定後、`インスタンスを起動` を選択する。

---

## 10.6 起動状態を確認する

EC2の一覧画面で、以下を確認する。

```text
インスタンスの状態: 実行中
ステータスチェック: 2/2のチェックに合格
パブリックIPv4アドレス: あり
プライベートIPv4アドレス: あり
サブネット: public-subnet-1a
セキュリティグループ: bastion-sg
```

# 12. Windows上で秘密鍵の権限を変更する

PowerShellを起動し、秘密鍵が保存されているディレクトリへ移動する。

```powershell
cd C:\Users\test\Downloads
```

秘密鍵が存在することを確認する。

```powershell
Get-ChildItem .\my-ec2-key.pem
```

現在ログインしているWindowsユーザーを確認する。

```powershell
whoami
```

表示例は以下である。

```text
desktop-xxxxx\pc_user
```

親フォルダから継承されているアクセス権限を削除する。

```powershell
icacls .\my-ec2-key.pem /inheritance:r
```

現在のWindowsユーザーへ読み取り権限を付与する。

```powershell
icacls .\my-ec2-key.pem /grant:r "DESKTOP-XXXXX\PC_User:(R)"
```

実際には、`whoami` で表示されたユーザー名を指定する。

現在の権限を確認する。

```powershell
icacls .\my-ec2-key.pem
```

# 13. PowerShellから踏み台サーバーへ接続する

PowerShellを起動し、秘密鍵が保存されているディレクトリへ移動する。

```powershell
cd C:\Users\test\Downloads
```

踏み台サーバーのパブリックIPv4アドレスを指定して接続する。

```powershell
ssh -i .\my-ec2-key.pem ec2-user@<踏み台サーバーのパブリックIPv4>
```




初回接続時は、以下の確認が表示される。

```text
Are you sure you want to continue connecting (yes/no/[fingerprint])?
```

接続先を確認し、以下を入力する。

```text
yes
```

以下のようなプロンプトが表示されれば接続成功である。

```text
[ec2-user@ip-10-0-1-xxx ~]$
```

# 14. Tera Termから踏み台サーバーへ接続する

Tera Termを起動する。

以下の内容を設定する。

```text
ホスト: 踏み台サーバーのパブリックIPv4
サービス: SSH
TCPポート: 22
SSHバージョン: SSH2
```

`OK` を選択する。

認証画面で以下の内容を設定する。

```text
ユーザー名: ec2-user
認証方式: RSA/DSA/ECDSA/ED25519鍵を使う
秘密鍵: C:\Users\test\Downloads\my-ec2-key.pem
```

設定後、`OK` を選択する。

以下のようなプロンプトが表示されれば接続成功である。

```text
[ec2-user@ip-10-0-1-xxx ~]$
```

# 16. 踏み台サーバー上に秘密鍵を作成する

踏み台サーバーへ接続した状態で、秘密鍵を保存するファイルを作成する。

```bash
vi my-ec2-key.pem
```

`vi` が起動したら、以下の操作を行う。

```text
1. iキーを押す
2. ローカルPCにある秘密鍵の内容を貼り付ける
3. Escキーを押す
4. :wq と入力する
5. Enterキーを押す
```

秘密鍵には、開始行と終了行を含めて貼り付ける。

```text
-----BEGIN PRIVATE KEY-----
秘密鍵の内容
-----END PRIVATE KEY-----
```

秘密鍵の形式によっては、以下の表示となる。

```text
-----BEGIN RSA PRIVATE KEY-----
秘密鍵の内容
-----END RSA PRIVATE KEY-----
```

# 17. 踏み台サーバー上の秘密鍵権限を変更する

秘密鍵の権限を、所有者だけが読み取れる状態へ変更する。

```bash
chmod 400 my-ec2-key.pem
```

権限を確認する。

```bash
ls -l my-ec2-key.pem
```

表示例は以下である。

```text
-r--------. 1 ec2-user ec2-user 1674 Jul 16 10:00 my-ec2-key.pem
```

# 20. 踏み台サーバーから秘密鍵を削除する

検証終了後、踏み台サーバーに保存した秘密鍵を削除する。

```bash
rm -f my-ec2-key.pem
```

秘密鍵が削除されたことを確認する。

```bash
ls -l my-ec2-key.pem
```

以下のように表示されれば削除されている。

```text
ls: cannot access 'my-ec2-key.pem': No such file or directory
```


今回は検証環境につきEC2 Instance Connectで接続する際は一時的に0.0.0.0を使用
